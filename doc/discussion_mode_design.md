# 讨论模式（Discussion Mode）设计文档

> 状态：**规划中，暂不实现**。本文记录设计共识，供未来动工时参考。
> 最后更新：2026-07-01

## 1. 一句话概括

新增一个独立于 Chat / Code 之外的**讨论模式**：让 2 个以上**不同的模型**（异构）就用户的问题，按「发散 → 深化 → 证伪 → 综合」的流程各司其职，最后由一个最强模型统合观点给出结论。

## 2. 为什么做（价值定位）

### 卖点是「结果」不是「过程」
- ❌ "我们的 app 能多 AI 讨论" —— 抽象能力，用户不 care，且默认背后是同一模型演多角色。
- ✅ "让 Claude 和 DeepSeek 就你的问题辩论后再综合" —— 具体模型 + 用户问题 + 承诺产出。用户只为「更好的答案」买单，不为「讨论」这个机制买单。

### 唯一的硬内核：异构（Heterogeneous）
- 同一个模型演多个角色 = 同一个大脑跟自己吵架，多样性是装出来的，收益有限（这是市面上同类功能「看着酷、用两次就腻」的根因）。
- gxAgent 天然具备**多 provider profile + provider 无关的消息层**，可以让 DeepSeek / Claude / Gemini 真的作为不同「专家」同台。不同模型的训练数据、偏好、盲区真实不同 —— 这是别的聊天 app 大多做不到的差异化，也是本功能有资格存在的门槛。

### 适用边界（重要）
- ✅ 适合：决策权衡、方案设计、写作、头脑风暴等**开放性、无唯一正解、受益于多视角**的问题。
- ❌ 不适合：编码、事实查询等有明确对错的任务 —— 那些需要「执行 → 验证」（即 Code 模式），不是「多模型商量」。
- **定位必须与 Code 模式泾渭分明，不追求通用。**

## 3. 核心流程（DAG 骨架）

单向流水线，每个节点职责不对称：

```
① 发散（Diverge）：提出各种开放性思路              —— 广度
② 深化（Deepen）：对这些思路展开详细论证           —— 深度
③ 反攻（Attack）：从方向上攻击上述论证，暴露漏洞      —— 证伪
④ 综合（Synthesize）：最强模型统合以上全部，给结论    —— 收敛
```

- **第③步「反方向攻击」是最值钱的一环**，也是多数同类产品缺失的：它强迫暴露漏洞，而非让观点越滚越自洽。
- **第④步综合是决定「惊艳 vs 鸡肋」的关键**：必须会取舍、指出分歧、给结论，而不是把观点拼接成又长又水的「正确的废话」。**动工时应在此投入最多打磨。**

### 单向流水线 vs 真·多轮（已决策）
- 本设计是「4 个角色各说一次」的**单向流水线**，不是来回多轮辩论。
- 单向：快、便宜、可控；但 ③ 攻击后即结束，①② 无机会回应。
- 真·多轮（①② 看到 ③ 的攻击后再反驳，来回 2-3 圈）：质量可能更高，但 token / 延迟翻几倍，且易发散收不住。
- **决策：先做单向版**（已含最有价值的「反攻」环节）。是否加多轮，留到用单向版跑过真实问题、确认「缺回合」确实是痛点后再定。

## 4. 数据模型：是「图」，但第一版不做「画布」

### 关键区分（最容易埋葬项目的坑）
- **图作为数据结构**：一个描述节点与边的 JSON + 执行引擎。**这是核心价值，成本可控。**
- **可视化节点编辑器（ComfyUI 那种连线画布）**：画布、拖拽、连线、环路/悬空边校验、序列化、撤销重做…… 是**一个独立产品，能吞掉数月**。而前端 `App.tsx` 已 5000+ 行。

### 决策：分层交付
```
先做：图的数据模型 + 执行引擎 + 2-3 个内置模板（JSON 定义的图）
      用户通过「选模板 + 换每个节点的模型/提示词」来用，不连线
后做：可视化编辑器（若确有用户要自己搭图，再上；可能永远不做）
```
- 理由：90% 用户会直接用模板，永远不碰连线。「文本讨论图」的节点种类少、复用价值低，不像图像生成那样撑得起画布的复杂度。
- **用户判断功能好坏看产出质量，不看能否自己连线。数据模型是图，不代表第一版就给用户一支画笔。**

### 节点（Node）概念性字段
- `id`
- `role` / `label`：如 diverge / deepen / attack / synthesize
- `model_profile`：绑定哪个 API profile（异构的落点）
- `prompt` / `preset_id`：该节点的立场提示词（复用现有 role preset 系统；也是借用 SillyTavern 预设的落点）
- `inputs`：引用哪些上游节点的输出（边的定义）
- `context_scope`：能看到哪些历史（见 §6 footgun 1）

### 边（Edge）
- 有向：`from_node -> to_node`，语义为「把 from 的输出喂给 to」。

## 5. 与现有架构的关系（实现其实不重）

现有底子几乎正好复用：
- **异构模型**：API profile 已能切换，每个节点绑一个 profile。
- **角色立场**：role preset 系统现成，①②③④ 即四套预设提示词。
- **消息编排**：provider 无关的消息层已能规范化各 wire 输出；四步串联是现有 agent loop 的变体。
- **前端展示**：已有 tool-output 卡片、reasoning 折叠，天然适合把四个角色发言分块折叠展示。

**主要新增工作量：**
1. **编排层**：一个新的「讨论模式」执行循环，按图的拓扑顺序调用不同 profile，把上游输出喂给下游。
2. **④综合节点的提示词**：最需打磨。

## 6. 已决策的关键设计点

1. **第一版图形态**：固定模板 + 可换每节点的模型/提示词，**不做可视化连线**。
2. **追问/继续**：采用「便宜做法」—— 图正常跑完，若综合节点认为信息不足，产出的结论即为「我需要你先回答 X/Y」，并**并回普通会话**；用户回答后重跑或普通对话继续。**不做中途暂停 / 断点恢复**（那需要部分完成状态与恢复语义，很麻烦）。是否做，留到确认「用户常在讨论中途被卡」后再说。
3. **节点默认上下文**：默认只看「**原始用户问题 + 直接上游节点输出**」，而非任意勾选全部历史。任意引用历史作为后续高级选项。

## 7. Footgun 清单（动工时警惕）

1. **上下文/账单爆炸**：允许节点任意引用历史 = context 爆炸 + 账单爆炸。默认收窄（见决策 3），自由引用仅作高级选项。
2. **人格匹配别硬编码**：异构 + 人格匹配（如「GPT 适合没感情的角色」）是真加分，但应作为**模板里的建议默认值（数据）**，而非写死在代码逻辑里。用户 key 不够时能降级到单模型。
3. **综合环节和稀泥**：见 §3，④ 若只拼接观点就是失败。
4. **成本与延迟**：N 模型 × R 轮 = 普通对话的数倍 token（用户自付 API 费）+ 串行延迟。需在 UI 上让用户对成本有预期。

## 8. 落地路线（建议）

**先做薄的验证假设，不要一上来搭完整预设流程系统：**
1. 最小版：2 个异构模型 + 单向流程 + 1 个综合节点，流程固定、不可配。
2. 用真实开放性问题跑几次，**诚实评估**：产出是否真比「直接问最强模型一次」更好？好到值这个钱和等待？
3. 若过关 → 再加 SillyTavern 预设流程组、轮数可配、模型可选、更多模板。
4. 若不过关 → 省下造复杂系统的时间。
5. 可视化编辑器：只有在确有用户要自己搭图时才做（第二阶段，可选）。
